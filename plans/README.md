# Implementation Plans

All plans stamped against commit `ae467a2`.

These are **design + implementation plans for roadmap features**, not bug fixes. Each
adds net-new product value and was chosen because the data foundation already exists in
the codebase but the user-facing capability does not. Execute in the order below unless
dependencies say otherwise. Each executor: read the plan fully before starting, honor its
STOP conditions, and update your row when done.

> **Repo reality the executor must know up front (applies to every plan):**
> - **No test framework exists.** `package.json` has no `test` script; there are zero test
>   files in the repo. Do **not** invent a test runner. Verification gates in these plans are
>   `npm run lint`, `npx tsc --noEmit`, `npm run build`, SQL applied to a local Supabase, and
>   explicit manual smoke tests. "Write a unit test" is never a step here.
> - **SQL changes are versioned migrations** under `supabase/migrations/`, named
>   `YYYYMMDDNNNNNN_slug.sql`. They are applied with the Supabase CLI. The executor must NOT
>   apply migrations to the production/staging project — see each plan's STOP conditions.
> - **The Next app lives in `pnaa/`.** Run all `npm` commands from `pnaa/`.

## Recommended order & why

1. **001 (Goals)** first — smallest, lowest-risk, fully self-contained. Good warm-up that
   establishes the new-table + new-RPC + dashboard-widget pattern the later plans reuse.
2. **002 (Trend snapshots)** next — **value compounds from the day capture starts**, and no
   backfill is possible (the daily job overwrites aggregates today). The sooner the snapshot
   cron lands, the more history the board eventually has. Land it early even though it shows
   little immediately.
3. **003 (CE certificates)** — highest org value, contained, but introduces a PDF-generation
   dependency decision (Step 1 is a spike).
4. **004 (Renewal reminders)** — biggest and riskiest of the original four: it requires
   provisioning an external email provider and coordinating with Wild Apricot's existing renewal
   emails. It has a human-decision gate the executor cannot resolve alone (STOP condition in the plan).
5. **005 (Member self-service portal)** — added in a follow-up. Low risk, mostly reuses
   `member-detail`'s rollup. Best done after **003** so it can host the self-download certificate
   button (soft dependency — it degrades gracefully if 003 isn't done yet).
6. **006 (Admin audit log)** — added in a follow-up. Lowest priority of the set; standalone
   governance feature. Fine to do any time.

## Dependency notes

- The plans are technically independent — none hard-imports another's code. The order above is by
  leverage and risk.
- **Migrations**: 001, 002, 004, and 006 each add a new Postgres table + RLS + migration; each picks
  the next free `YYYYMMDDNNNNNN_` migration filename. If executed out of order, keep migration
  timestamps monotonic (later-executed plan gets the later timestamp).
- **`pnaa/components/layout/sidebar.tsx` is touched by 001, 005, and 006** (each adds a nav item). If
  these run on separate branches, expect a small merge conflict in the nav array — trivial to resolve
  (each just adds one entry). 005 and 006 also note "if another plan already added role-conditioning
  to the nav, keep your item consistent with it."
- **003 ↔ 005**: certificates are the natural thing a member self-downloads from the portal. 003
  builds the data/admin side; 005 mounts 003's `<CertificateButton>` if it exists, and skips it
  otherwise. Do 003 first for the full experience, but neither blocks the other.
